Introduction

Starting a new Antora project compatible with Modular Documentation

Starting a new Antora project compatible with Modular Documentation.

Prerequisites

  • Docker or Podman.

Procedure

  1. Create a repository using the Antora for Modular Docs repository template and clone the repository to your local environment.

  2. Alternatively, download the Antora for Modular Docs zip archive, decompress it, and use the content of this archive to start a new repository.

    $ cd antora-for-modular-docs
$ git init

  3. The new repository contains:

    • The current documentation.

    • Antora configuration, to adapt to your needs.

    • A GitHub workflow to publish the docs using GitHub Pages.

    • A GitHub workflow to build, validate and publish a preview for pull requests.

    • Configuration to edit and build the documentation on Eclipse Che hosted by Red Hat

    • Configuration for the htmltest link checker and other tools.

  4. Current example defines an unique module: end-user-guide. Adapt the setup to your needs:

    • Optionally, rename the end-user-guide directory.

      Don’t use the default ROOT module name, always use a Named Module Directory.

    • Optionally, add a new module directory, and create the requested subdirectories. See Antora - Standard File and Directory Set.

  5. Edit antora.yml. See What’s antora.yml?

    antora.yml
    ---
# Antora component version descriptor.
name: <docs> (1)
title: <Component Title> (2)
version: <master> (3)
display_version: <Stable version> (4)
start_page: <end-user-guide:introduction.adoc> (5)
nav:
  - <modules/end-user-guide/nav.adoc> (6)
asciidoc:
  attributes:
    # for antora
    experimental: ""
    favicon: favicon.png
    icons: font
    # for the project
    context: <docs> (7)
    1 Mandatory component name with no spaces: results in an URL fragment. See Antora - name key. Example: docs.
    2 Optional component title. See Antora - Customize the Title
    3 Mandatory version name. See Antora - version key. Set to master to remove the version from the page URL, see Antora - Define a Component Version with No Version.
    4 Optional version display name. Example: Stable. See Antora - Customize the Display Version
    5 Define the start page. See: Antora - Choose a Start Page
    6 Define navigation file. See: Antora - Assign Navigation Files to a Component Version
    7 Define project context. (no spaces: results in an URL fragment) See: Modular Docs - File Names and Anchors

  6. Edit antora-playbook-for-development.yml. See What’s an Antora playbook?

    antora-playbook-for-development.yml
    ---
# Antora Playbook for development.
# Use it to build uncommitted changes.
#
site:
  title: <Component Name Documentation> (1)
  # url: Disabled on purpose to avoid htmltest crawling the live website.
  start_page: <docs>::index.adoc (2)
  robots: allow
content:
  sources:
    - url: ./
      branches: HEAD
output:
  destinations:
    - provider: fs
      clean: true
      path: build/site
ui:
  bundle:
    url: https://gitlab.com/antora/antora-ui-default/-/jobs/artifacts/master/raw/build/ui-bundle.zip?job=bundle-stable
    snapshot: true
  supplemental_files: ./supplemental-ui
urls:
  html_extension_style: indexify
  redirect_facility: static
runtime:
  cache_dir: ./.cache/antora
    1 Define project title.
    2 Define project start_page.

  7. Edit antora-playbook.yml

    antora-playbook.yml
    ---
# Use this Antora Playbook for publication, to build the released version branch.
site:
  title: <Component Name Documentation> (1)
  # The url property is mandatory to generate the 404 and sitemap files.
  url: https://project.tld (2)
  start_page: <docs>::index.adoc (3)
  keys:
    google_analytics: "<google_analytics_key>" (4)
  robots: allow
content:
  sources:
    - url: ./
      branches:
        - stable (5)
        - main (5)
      edit_url: "<git_repository_URL>/edit/main/{path}" (6)
output:
  destinations:
    - provider: fs
      clean: true
      path: build/site
ui:
  bundle:
    url: https://gitlab.com/antora/antora-ui-default/-/jobs/artifacts/master/raw/build/ui-bundle.zip?job=bundle-stable
    snapshot: true
  supplemental_files: ./supplemental-ui
urls:
  html_extension_style: indexify
  redirect_facility: static
runtime:
  cache_dir: ./.cache/antora
    1 Define project title.
    2 Define peoject url
    3 Define project start_page.
    4 Define google_analytics key.
    5 Define branches to build.
    6 Define edit_url.

  8. Replace supplemental-ui/img/project-icon.svg by the project logo.

  9. Optionally, edit supplemental-ui/partials/header-content.hbs.

  10. Create an initial commit containing the result and publish to the remote Git repository.

    $ git add .
$ git commit
$ git push

  11. To publish the docs using GitHub pages, enable GitHub Pages in the remote repository configuration: choose gh-pages as source branch, and root as source directory.

  12. To publish the build result for the pull requests:

    1. Create a Netlify token NETLIFY_AUTH_TOKEN. See: Netlify - Create a new personal access token. See: link: Obtaining a token with the UI.

    2. Get a Netlify site ID NETLIFY_SITE_ID. See: Link with an environment variable.

  13. Set the NETLIFY_AUTH_TOKEN and NETLIFY_SITE_ID environment variables in the remote git repository configuration (settings/secrets/actions). See: Netlify GitHub Action CLI documentation

Verification

  1. Run preview on a local environment

    ./runnerpreview.sh

  2. Navigate to the remote git repository and watch for the changes in the Actions and Pages.
